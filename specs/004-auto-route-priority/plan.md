# Implementation Plan: Automatic Camera-Priority Routing

**Branch**: `004-auto-route-priority` | **Date**: 2026-06-09 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/004-auto-route-priority/spec.md`

## Summary

Remove the three-way avoidance preference UI (avoid / balanced / fastest) and replace it with a single automatic routing strategy: attempt a zero-camera route first; if the routing engine cannot satisfy strict avoidance, fall back to the fastest available route and report the remaining camera exposure. The backend always uses the existing "avoid" code path; the preference parameter is removed from the API and all frontend state management is eliminated.

## Technical Context

**Language/Version**: Ruby 3.4.x (backend), TypeScript 5.x (frontend)

**Primary Dependencies**: Rails 8.1.x (API mode), React 19, Vite, Valhalla (routing), MapLibre GL JS v5

**Storage**: PostgreSQL 16 + PostGIS (no schema changes)

**Testing**: RSpec (backend), Vitest + React Testing Library (frontend unit), Playwright (frontend e2e)

**Target Platform**: Linux server (production), Docker Compose (dev)

**Project Type**: Full-stack web application (Rails API + React SPA)

**Performance Goals**: Route planning p95 ≤ same as current (≤ 2 Valhalla calls per request — unchanged from existing "avoid" mode)

**Constraints**: Valhalla supports only hard polygon exclusion; true minimum-camera routing is not natively available. The fallback is the fastest route, not the provably fewest-cameras route. (See research.md.)

**Scale/Scope**: Single-page app; no multi-user/concurrency concern for this change
